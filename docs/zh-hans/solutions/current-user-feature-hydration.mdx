# 1. 功能定位

当前用户上下文承担两类职责：

* 基础上下文：登录用户、租户、角色权限、员工信息、组织成员关系和当前作用域。
* Feature 上下文：租户级 feature、组织级 feature，以及依赖这些 feature 控制入口、路由和页面能力的状态。

Feature 上下文 Hydration 的目标，是把这两类数据从加载时机和缓存策略上拆开：

* 应用启动优先拿到基础上下文，保证页面可以尽快进入可用状态。
* feature 数据在后台或路由准入阶段补齐，保证功能开关判断仍然准确。
* 数据变更后需要刷新用户上下文时，可以明确区分“完整用户上下文刷新”和“feature 上下文刷新”。
* 后端缓存只覆盖 feature 专属数据，避免组织名称、权限、员工信息等基础字段被缓存成旧数据。

---

# 2. 设计目标

这套机制主要解决四类问题：

* `/user/me` 首屏返回数据过大，影响应用初始化速度。
* ClawXpert 等 feature 关闭后，隐藏入口但仍可能通过 URL 直访。
* 组织新增、删除、默认组织切换或基础信息更新后，前端用户上下文可能仍保留旧组织数据。
* feature 数据可以缓存，但不能把完整用户数据结构一起缓存，避免权限、员工、组织基础字段变旧。

设计上的基本原则：

* 基础上下文优先，feature 上下文延后补齐。
* 受 feature 控制的入口以最新 feature hydration 结果为准。
* 组织管理类数据变更后使用完整用户上下文刷新。
* 缓存粒度必须区分 tenant、user、请求 shape 和失效版本。
* 失败或超时时采用安全优先策略，默认不放行受 feature 控制的入口。

---

# 3. 请求分层

## 3.1 基础上下文请求

应用启动阶段只加载进入系统必须依赖的用户上下文，包括：

* 当前用户基础信息
* 当前租户
* 当前用户角色和权限
* 员工信息
* 当前用户所属组织
* 默认组织和 active scope 初始化所需数据

这部分请求会参与应用初始化流程，因此只保留首屏必需数据。

## 3.2 Feature 上下文请求

Feature 上下文在基础上下文之后补齐，主要包含：

* 租户级 feature 配置
* 组织级 feature 配置
* feature 对应的元信息
* 当前选中组织可用的 feature 列表

Feature 上下文可以由后台加载，也可以在进入受控路由前加载。

页面不需要 feature 判断时，不应该被 feature hydration 阻塞。

## 3.3 状态合并

Feature hydration 返回后，只把 feature 相关数据合并回当前用户上下文：

* 更新租户级 feature 列表。
* 更新组织级 feature 列表。
* 保留基础上下文中已有的用户、权限、员工、组织基础字段。
* 根据当前选中组织同步当前组织 feature 状态。

这样可以避免 feature hydration 覆盖掉基础上下文中的新组织、新权限或新员工信息。

---

# 4. 前端状态语义

Feature 上下文有三个核心状态：

* 已完成：feature 上下文已经可用于功能开关判断。
* 加载中：feature 上下文正在请求或等待复用中的请求。
* 已失败：feature 上下文请求失败，受 feature 控制的入口按不可用处理。

状态收敛规则：

* hydration 成功后，标记为已完成，并清除加载中和失败状态。
* 命中浏览器会话缓存时，也视为一次成功 hydration，并清除旧的加载中和失败状态。
* 前台请求失败时，标记失败并结束加载。
* 后台请求失败时，不阻断当前页面，但会保留失败状态供受控入口判断。

为了避免重复请求，前端会复用同一用户、同一租户、同一请求 shape 下正在进行的 hydration。

数据变更后的强制刷新不复用旧请求，确保不会拿到变更之前发出的旧响应。

# 5. 路由准入

## 5.1 默认聊天入口

访问 `/chat` 默认入口时，系统会先尝试拿到 feature 上下文，再决定进入哪个聊天入口：

* feature hydration 成功，且 ClawXpert 所需 feature 都开启：进入 ClawXpert。
* feature hydration 失败：进入通用助手。
* feature hydration 超过 3 秒仍未完成：进入通用助手。
* ClawXpert 所需 feature 未开启：进入通用助手。

这里采用安全优先策略：不能确认 feature 已开启时，不自动进入受控功能。

## 5.2 ClawXpert 直访

ClawXpert 页面不仅依赖侧边栏入口控制，也依赖路由准入控制。

用户直接访问 ClawXpert URL 时，系统会先完成 feature 判断：

* feature 已确认开启：允许进入。
* feature 已确认关闭：跳回聊天入口。
* hydration 失败或 3 秒超时：跳回聊天入口。

这保证“入口隐藏”和“URL 直访”使用同一套 feature 判断。

## 5.3 已打开页面的状态变化

当用户已经停留在 ClawXpert 页面时，如果 feature hydration 后发现 ClawXpert 不再可用，页面会主动离开受控页面并回到通用助手。

这用于覆盖 active route 不重新触发路由守卫的情况。

# 6. 组织上下文一致性

组织管理页的新增、删除、更新和默认组织切换，会影响的不只是 feature 数据，还包括当前用户完整组织成员关系。

因此这类数据变更成功后需要刷新完整用户上下文，而不是只刷新 feature 上下文。

刷新后需要同步：

* 当前用户的组织成员关系
* 当前租户 feature
* 当前选中组织
* 当前选中组织 feature
* 组织详情页的选中对象和表单数据

当当前选中组织已经不存在，或当前用户不再属于该组织时，选择策略是：

1. 优先切到默认组织。
2. 没有默认组织时切到第一个可用组织。
3. 没有任何组织时清空当前组织和组织 feature。

这保证以下操作完成后，页面和全局用户上下文都立即一致：

* 新增组织
* 删除组织
* 修改组织名称或基础信息
* 切换默认组织
* 修改组织启用状态

# 7. 后端 Feature Context Cache

## 7.1 缓存内容

后端只缓存 feature 专属上下文，不缓存完整 current user。

缓存内容包括：

* 租户级 feature 列表
* 当前用户所属组织的组织级 feature 列表

这些数据会在请求时附加回当前用户上下文。

以下数据每次仍按当前用户上下文重新读取，不进入 feature context cache：

* 用户基础字段
* 员工信息
* 角色和权限
* 租户基础字段
* 组织基础字段
* 组织成员关系

这样可以让 feature context 有短期缓存收益，同时避免组织名、权限、员工信息等基础数据出现缓存旧值。

## 7.2 缓存粒度

后端 cache 不是简单的“一个 user 一个缓存”。

feature context 数据缓存粒度包含：

* tenantId
* userId
* 请求关系集合的 hash
* tenant/user feature cache version

也就是说，同一个用户在不同租户下会有不同缓存；同一个用户请求不同 feature context shape 时也会有不同缓存。

## 7.3 Redis key 设计

Redis 中实际有三类 key，统一前缀是：

```text
user:me:feature-context
```

真正保存 feature context 数据的 key：

```text
user:me:feature-context:v1:{tenantId}:{userId}:{relationsHash}:{version}
```

其中 `version` 由 tenant version 和 user version 拼成：

```text
{tenantVersion}.{userVersion}
```

用于失效的 version key：

```text
user:me:feature-context:tenant-version:{tenantId}
user:me:feature-context:user-version:{tenantId}:{userId}
```

示例：

```text
user:me:feature-context:v1:tenant-1:user-1:abc123def4567890:1714460000000.1714460000123
user:me:feature-context:tenant-version:tenant-1
user:me:feature-context:user-version:tenant-1:user-1
```

## 7.4 失效策略

feature 配置变更后，更新 tenant version。

用户组织成员关系变更后，更新 user version。

新的 version 会生成新的 feature context 数据 key，旧 key 不再被命中，等待 TTL 自然过期。

这样可以避免主动批量扫描和删除 Redis key，同时保证数据变更后读取到新的 feature context。

---

# 8. 行为保障

这套功能需要保证以下行为：

* 应用启动不被完整 feature 关系阻塞。
* ClawXpert feature 关闭后，入口隐藏且 URL 直访也无法进入。
* `/chat` 默认入口在 feature hydration 失败或超时时进入通用助手。
* 首次进入 ClawXpert 时，路由判断不会重复触发多次用户上下文请求。
* 数据变更后的强制刷新不会复用变更前未完成的 hydration 请求。
* 同一用户切换 tenant 后，不复用旧 tenant 的 hydration 请求、已完成标记或缓存。
* 浏览器会话缓存命中后，loading 和 failed 状态会被正确清理。
* 当前选中组织被删除或用户不再属于该组织时，不保留旧组织 feature。
* 新增、删除、更新组织后，全局用户上下文和组织详情页立即同步。
* 修改组织、权限、员工信息后，不会被 feature context cache 返回旧值。
